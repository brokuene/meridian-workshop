Our Approach

Stabilise. Unlock. Build. Enhance.

We work in dependency order — not requirement order. You can't safely build new features on a broken, untested foundation.

Phase 1

Stabilise

Fix all Reports defects (R1). Document the architecture for IT (R4). These run in week 1–2 while the rest of the team ramps.

Phase 2

Unlock

Establish Playwright end-to-end test coverage (R3). This is IT's gate — nothing else gets approved without it. Tests are written as we fix.

Phase 3

Build

Deliver the Restocking view (R2) — new backend endpoint + frontend — on a verified, tested foundation. The right order matters.

Phase 4

Enhance

UI refresh (D1), full Japanese i18n (D2), dark mode (D3) — parallel workstreams. Done alongside Build; do not block the required delivery.

Requirement R1

Reports Module Remediation

We found the defects ourselves. We didn't wait for a bug list.

  • No i18n — all text hardcoded, existing composable unused
  • Manual formatters duplicating locale utilities
  • Debug console.log() calls in production
  • No error retry on failed data load
  • Calculation logic inline in view — fragile, untestable

Approach

Full defect audit → fix by category → wire existing useI18n() composable → replace manual formatters → extract computed logic → verify against live endpoints.

No new infrastructure. Everything we need is already in the codebase.

Requirement R2

Restocking Recommendations

The feature doesn't exist yet — but most of the data does. Inventory levels, demand forecasts, and purchase order models are already in the backend. We wire them together.

New: /api/restocking

Accepts warehouse filter + budget ceiling. Cross-references stock vs. demand threshold. Returns ranked purchase order recommendations within budget.

New: RestockingView.vue

Warehouse selector, budget ceiling input, recommendations table, running total indicator. Wired into router and nav.

Logic model
inventory stock → below reorder threshold?
demand forecast → how much is needed?
budget ceiling → what fits?
