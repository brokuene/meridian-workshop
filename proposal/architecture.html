1 System Overview

The system is a two-tier web application: a Vue 3 single-page application served by Vite on port 3000, communicating with a FastAPI Python backend on port 8001 via REST/JSON. All persistent data is loaded from JSON files into memory at startup. There is no database layer in the current deployment.

Backend API
Data files (JSON, read-only)
No persistence — data is in-memory
Production note: All data lives in JSON files and is loaded into process memory at startup. There is no database. Any writes (purchase orders, tasks) are lost when the server restarts. A database migration is the recommended first step before production use.

2 Frontend Architecture

The frontend is a Vue 3 Composition API application. All pages are single-file components (.vue) in client/src/views/. Shared logic lives in composables. No external UI library is used — charts are hand-built SVG.

Filter data-flow

All four global filter values (Time Period, Location, Category, Order Status) are held in a singleton useFilters() composable. FilterBar writes to it; views read from it and re-fetch via api.js when values change.

StepActorAction
1FilterBar.vueUser selects a value → writes to useFilters() ref
2useFilters()Reactive ref updates; all watchers notified
3View componentwatch(selectedLocation, loadData) triggers
4View → api.jsgetCurrentFilters() called → params object built
5api.jsURLSearchParams constructed; 'all' values omitted; axios GET fired
6FastAPIQuery params received; in-memory data filtered; JSON returned
7View componentref(data).value updated → Vue re-renders

3 Backend Architecture

The backend is a single-file FastAPI application (server/main.py) with a companion data-loader module (server/mock_data.py). All data is loaded once at startup and held in module-level Python lists for the lifetime of the process.

Startup sequence

StepWhat happens
1uvicorn main:app imports main.py
2mock_data.py is imported — load_json_file() reads every data/*.json into memory
3Module-level variables (inventory_items, orders, …) are populated — they stay constant for the entire session
4FastAPI() instance created, CORS middleware added (allow_origins=["*"])
5All @app.get / @app.post decorators register routes
6Uvicorn starts listening on 0.0.0.0:8001

Internal structure of main.py

Shared filter utilities

Two helper functions in main.py centralise all filtering logic. Every endpoint that accepts filter params calls these rather than re-implementing the logic.

FunctionParamsLogic
apply_filters(items, warehouse, category, status) All optional Sequential list comprehensions; skips filter if value is None or 'all'; case-insensitive on category/status
filter_by_month(items, month) month: YYYY-MM or Q1–Q4 string If month starts with Q, expands to the three constituent months via QUARTER_MAP; otherwise substring-matches order_date

6 JSON Storage Layer

All persistent data lives in seven JSON files under server/data/. They are read once at startup by mock_data.py and held as Python lists/dicts for the process lifetime. No writes are persisted — restarting the server reloads the originals.

FileRecordsTop-level shapePrimary keyNotes
inventory.json 32 Array of objects id (string "1"–"32") One row per SKU per warehouse; location is shelf code, warehouse is city
orders.json 250 Array of objects id (string) Each order embeds its line items as a nested array; warehouse and category are denormalised onto the order header
demand_forecasts.json 9 Array of objects id (string) One forecast per SKU; trend is increasing / stable / decreasing; period is "Next 30 days" for all current rows
backlog_items.json 4 Array of objects id (string) Links to orders via order_id; has_purchase_order is computed at runtime by joining against purchase_orders
purchase_orders.json 0 (empty) Array of objects id (string) New POs created via POST /api/purchase-orders are appended here in memory only — file stays empty; links to backlog via backlog_item_id
spending.json 1 object Single object with three keys — spending_summary (annual totals), monthly_spending (12-item array), category_spending (4-item array) — all USD, calendar year 2025
transactions.json 56 Array of objects id (string) Recent ledger transactions; not filterable via API — always returned in full

Entity relationships

Key referential notes

RelationshipEnforced?Detail
demand_forecasts.item_sku → inventory.sku Soft ref No FK constraint; the 9 forecast SKUs are a subset of the 32 inventory SKUs. The restocking algorithm joins them in Python at query time.
backlog_items.order_id → orders.id / order_number Soft ref Dashboard joins backlog to orders by order_id to show shortage context. No constraint validation.
purchase_orders.backlog_item_id → backlog_items.id Soft ref The /api/backlog endpoint computes has_purchase_order by scanning purchase_orders for matching backlog_item_id.
orders.items[].sku → inventory.sku Denorm Line-item SKUs are embedded inside each order object. No live join performed — order data is self-contained.
Updating data: Edit the relevant JSON file, then restart the server (uvicorn main:app --reload). Validate JSON structure before committing — a malformed file will crash the import and the server will not start.

8 Key Patterns for Future Maintainers

Adding a new view

  1. Create client/src/views/MyView.vue using Composition API + useFilters() + useI18n()
  2. Add route in client/src/main.js
  3. Add nav link in client/src/App.vue
  4. Add translation keys to src/locales/en.js and src/locales/ja.js

Adding a new API endpoint

  1. Define a Pydantic model in server/main.py
  2. Add @app.get("/api/...") function with Optional[str] query params
  3. Add corresponding method to client/src/api.js

Internationalization

All user-visible strings should go through t('key.path'). Add the key to both en.js and ja.js before using it in a template. Product names, customer names, and warehouse names have dedicated translation maps in each locale file.
